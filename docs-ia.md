# SafeDep Documentation — Information Architecture

> **Status:** Living document · **Canonical home:** this file (mirrored to Notion — see `docs-ia-proposal.md` for the Notion link) · **Rationale & benchmarks:** see `docs-ia-proposal.md`
>
> This is the **canonical, maintained** description of how the SafeDep docs are organized and how new documentation must be placed. The proposal doc records *why* we chose this structure (benchmarks, alternatives, migration map); this doc records *what the structure is* and *how to extend it*. When the structure changes, update **this file, the Notion mirror, and the condensed copy in `CLAUDE.md`**.

## 1. The structure

Six goal-first tabs — **hybrid axis**: goals at the top, products nested one level deeper, persona on-ramps on the homepage.

| Tab | Purpose | Primary audience | Groups |
|---|---|---|---|
| **Get Started** | "What is SafeDep?", route by persona, define vocabulary | All | Introduction · Quickstarts · Core Concepts *(backlog)* |
| **Protect Developers** | Block threats on the dev machine and in AI coding tools | Developer | Package Manager Guard (pmg) · AI Coding Protection & Audit · Endpoint Sync |
| **Scan & Analyze** | Find supply-chain risk in code, deps, SBOMs, CI/CD | Developer / DevOps | Repository Scanning (vet) · Bill of Materials (xBom) · AI Governance · CI/CD & Platform Integrations |
| **Govern & Manage** | Operate SafeDep Cloud: visibility, policy, access | Security Engineer | SafeDep Cloud · Endpoint Hub · Policy & Risk · Access & Identity |
| **Reference** | Look up exact syntax, APIs, flags | Sec Eng / Developer | vet Query & Policy · API & Automation |
| **Community & Support** | Get help, join the community | All | Community · Support |

## 2. Governing rules (how the IA evolves)

Every structural decision derives from these. The rationale for each is in `docs-ia-proposal.md` §2.

- **R1 — One primary entry point.** New visitors meet the "What is SafeDep?" primer and a persona choice, not a tool catalog.
- **R2 — Two audiences, both at home.** Serve the developer (installing tools locally) and the security engineer (dashboards/policy) without forcing either through the other's content.
- **R3 — Organize by goal, not by tool.** Tabs name what the user wants to do; tool-specific material lives one level deeper.
- **R4 — Concepts are first-class, atomic, linkable.** Each concept (Policy, Endpoint, Tenant, Malysis/Malbase, SBOM, CEL) gets one dedicated page under Get Started › Core Concepts.
- **R5 — Progressive disclosure.** Top level is opinionated and narrow; exhaustive reference lives in the Reference tab.
- **R6 — AI-agent consumability is first-class.** One concept per atomic page, predictable headings, stable URLs. `llms.txt`/`llms-full.txt` are auto-generated by Mintlify.

**The test for any new page:** *What goal does this serve, and for whom?* If the answer is "it documents tool X" rather than "it helps a developer/security-engineer do Y," it is in the wrong place.

## 3. Where does a new page go?

| If the page... | Tab | Example group |
|---|---|---|
| Defines a concept/term used across docs | Get Started | Core Concepts |
| Onboards a brand-new user / routes by persona | Get Started | Introduction / Quickstarts |
| Helps block threats on a dev machine or in AI coding tools | Protect Developers | pmg / AI Coding Protection & Audit / Endpoint Sync |
| Helps find risk in code, deps, SBOMs, or CI/CD | Scan & Analyze | Repository Scanning / Bill of Materials / AI Governance / CI/CD & Platform Integrations |
| Covers operating SafeDep Cloud (visibility, policy, access) | Govern & Manage | SafeDep Cloud / Endpoint Hub / Policy & Risk / Access & Identity |
| Is pure lookup (syntax, flags, API) | Reference | vet Query & Policy / API & Automation |
| Is community/support | Community & Support | Community / Support |

A page appears in **exactly one** sidebar slot. To surface it from another tab, add an in-body `<Card>` or link — never a second nav entry.

## 4. File path convention for NEW pages

The sidebar is goal-based, but many existing files still live at legacy tool-based paths (`vet/`, `pmg/`, `xbom/`, `cloud/`, `apps/`). Those are **grandfathered** — do not move them ad hoc (moving a file changes its URL and requires a redirect).

**New pages MUST use goal/topic-based paths**, matching the IA:

| Path prefix | Used for |
|---|---|
| `get-started/` | Primer, persona routing, quickstart aggregators |
| `concepts/` | Atomic concept pages (e.g. `concepts/policy.mdx`) |
| `protect/` | Goal-section landings and net-new Protect Developers content |
| `scan/` | Goal-section landings and net-new Scan & Analyze content |
| `govern/` | Goal-section landings and net-new Govern & Manage content |
| `reference/` | Net-new Reference content |

A later "path-alignment" pass may migrate legacy paths to goal-based ones; that pass will add a `redirects` block in `docs.json` (`{ "source": "/old", "destination": "/new" }`, permanent 308 by default) so external/bookmarked URLs never 404.

## 5. Naming, frontmatter, and Diátaxis

**File names:** kebab-case, specific (`github-code-scanning.mdx`, not `guide.mdx`).

**Frontmatter (required for every page):**
```yaml
---
title: Descriptive Page Title
description: "Clear, searchable description under 160 characters"
---
```
No H1 — Mintlify renders the title from frontmatter.

**One Diátaxis type per page:**

| Diátaxis type | Style | Where it lives |
|---|---|---|
| **Tutorial** (quickstarts) | Learning-oriented, hands-on, safe to fail | Get Started › Quickstarts |
| **How-to Guide** | Problem-oriented, step-by-step, practical | The relevant goal tab (Protect / Scan / Govern) |
| **Reference** | Information-oriented, minimal explanation, comprehensive | Reference tab |
| **Explanation** (concepts) | Understanding-oriented, theoretical, connects concepts | Get Started › Core Concepts |

Don't mix Diátaxis types in a single page.

## 6. How to change the IA

When you add/rename/move a tab or group, or change a placement rule:

1. Update `docs.json` navigation (the `tabs` array).
2. Update **this file** (`docs-ia.md`) — the §1 structure table and any changed rule.
3. Update the **Notion mirror** (same content, same headings).
4. Update the **condensed copy** in `CLAUDE.md` so the AI-facing rules stay in sync.
5. If a file's on-disk path changes, add a `redirects` entry in `docs.json`:
   ```json
   { "source": "/old-path", "destination": "/new-path" }
   ```
   Verify with `node_modules/.bin/mintlify broken-links` after every change.
