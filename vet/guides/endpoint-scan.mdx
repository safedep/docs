title: Endpoint Scan
description: "Inventory AI tools, MCP servers, coding agents, IDE extensions, and agent skills on developer endpoints"
---

`vet endpoint scan` performs a local inventory of AI tooling installed on the current machine or
project directory. It discovers MCP server configurations, coding agents, CLI tools, IDE extensions,
project configuration files, and agent skill directories â€” giving security and platform teams
visibility into what AI tooling is active across developer endpoints.

Results are printed as a table and optionally written to a JSON file. When SafeDep Cloud credentials
are configured, discovered items are streamed to the cloud for centralized tracking.

## Prerequisites

- `vet` [installed](/vet/quickstart)
- For cloud sync: SafeDep Cloud credentials configured via `vet auth configure`

## Usage

Scan the current endpoint for all AI tooling:

```bash
vet endpoint scan
```

### Scope Filtering

Restrict discovery to system-level or project-level signals:

```bash
# System-level only (global configs, CLI binaries, IDE extensions)
vet endpoint scan --scope system

# Project-level only for a specific repository
vet endpoint scan --scope project -D /path/to/repo
```

### Kind Filtering

Restrict to a specific scanner kind:

```bash
# Only AI tools (MCP servers, agents, CLI tools, extensions, project configs)
vet endpoint scan --kind ai-tool

# Only agent skill directories
vet endpoint scan --kind agent-skill
```

### JSON Output

Write a structured JSON inventory for downstream processing:

```bash
vet endpoint scan --report-json inventory.json

# JSON only, suppress table output
vet endpoint scan --report-json inventory.json --silent
```

## Scanner Kinds

The command runs two independent scanners, each covering a different class of signals.

| Kind           | What it finds                                                              |
|----------------|----------------------------------------------------------------------------|
| `ai-tool`      | MCP servers, coding agents, CLI tools, IDE extensions, project config files |
| `agent-skill`  | Agent skill directories inside well-known skill paths                      |

## Item Types

Every discovered item is classified into one of the following types:

| Type             | Description                                                                          |
|------------------|--------------------------------------------------------------------------------------|
| `mcp_server`     | Model Context Protocol server entry found in a config file                           |
| `coding_agent`   | Coding agent detected via a system-level config directory or installed binary        |
| `cli_tool`       | AI CLI binary found on `$PATH`, version-verified by executing with a version flag    |
| `ai_extension`   | AI-related IDE extension detected from installed extension manifests                 |
| `project_config` | AI tool instruction or config file found in a project repository                     |
| `agent_skill`    | Skill subdirectory found inside a well-known agent skills path                       |

## Scopes

| Scope     | Base path                                          |
|-----------|----------------------------------------------------|
| `system`  | User home directory â€” global configs and binaries  |
| `project` | Project directory (default: cwd, override with `-D`) |

### Project Config Files

### CLI Tools

AI CLI binaries are discovered by searching `$PATH` for known names. Each candidate is executed
with a version flag and its output is matched against a known pattern to confirm it is the expected tool.

| Tool                    | Binary(s)              | Verification                                   |
|-------------------------|------------------------|------------------------------------------------|
| Claude Code CLI         | `claude`               | Output matches `claude v<semver>`              |
| Cursor CLI              | `cursor`               | Semver on first line of `--version`            |
| Windsurf CLI            | `windsurf`             | Semver on first line of `--version`            |
| Antigravity / Gemini CLI | `antigravity`, `ag-kit` | Semver on first line of `--version`           |
| VS Code CLI             | `code`                 | Semver on first line of `--version`            |
| Aider                   | `aider`                | Output contains `aider v<semver>`              |
| GitHub Copilot          | `gh`                   | `github/gh-copilot` in `gh extension list`    |
| Amazon Q                | `q`, `amazon-q`        | Output contains `amazon` or `aws` + semver     |

## MCP Server Details

For each discovered MCP server, the following fields are captured:

| Field              | Description                                                          |
|--------------------|----------------------------------------------------------------------|
| Transport          | `stdio`, `sse`, or `streamable_http`                                 |
| Command (stdio)    | Binary path and arguments                                            |
| URL (HTTP)         | Server endpoint URL                                                  |
| EnvVarNames        | Names of environment variables referenced (values not captured)      |
| HeaderNames        | Names of HTTP headers referenced (values not captured)               |
| AllowedTools       | Explicit tool allowlist if configured                                |
| AllowedResources   | Explicit resource allowlist if configured                            |

## Cloud Sync

When SafeDep Cloud credentials are available, all discovered items are streamed to SafeDep Cloud
over gRPC in addition to local output. A local write-ahead log buffers undelivered events across
restarts (capped at 100k items). The connection drains on exit up to `--drain-timeout` (default: 30s).

Configure credentials:

```bash
vet auth configure
```

Or set environment variables:

```bash
export SAFEDEP_API_KEY=<key>
export SAFEDEP_TENANT_ID=<tenant>
```

## Security

The scan never captures environment variable or header values â€” only key names are recorded.
No network calls are made during discovery. All inspection is local filesystem and `$PATH` based.
